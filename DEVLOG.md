# MdExplorer 開発日誌

AIとの協働開発における実装履歴と振り返り。
各機能について「当初の実装で抜けていた点」と「AIとして不足していた視点」を記録する。

---

## 実装履歴

### 1. Mermaid ダイアグラムのズーム・パン

CSS transform + wheel/drag イベントで SVG をズーム・パン可能にした。

**当初の抜け**
- SVG が存在しない（エラー状態など）場合にコントロールが表示されたままになった。
- SVG の存在確認を後から追加。

---

### 2. ファイルスキャンのパフォーマンス改善

34,000 件超のフォルダで開くまで約 5 秒かかっていた。

**原因と対処**
- `scanDir()` 内のサブディレクトリ走査が逐次 `await` になっており O(N) の直列 I/O になっていた。
- `Promise.all()` による並列化と、フロントエンドで未使用だったファイルごとの `stat()` 呼び出し削除（34,000 回 → 0 回）で約 1 秒まで短縮。

**AI として不足していた視点**
- 初期実装時にスキャンが逐次処理になっていることに気づかなかった。パフォーマンスの問題はユーザーが実際に使って初めて顕在化した。大量ファイルを想定した負荷テストの視点が欠けていた。

---

### 3. フォントサイズ制御

ヘッダーの A- / A+ ボタンでサイドバー・プレビュー・タブ・タグ欄などすべての表示サイズを連動して変更。

**当初の抜け**
- CSS 変数 `--preview-font-size` を `#file-view` に適用していたが、タブバーやサイドバーが連動していなかった。`#content` と `#sidebar` に移して解決。
- ライトモードでフォントサイズラベルが背景色と同化して不可視になった。`color: var(--text-header)` を後から追加。

---

### 4. タグ機能の拡張（大文字保持・編集・フィルタバー）

タグの大文字/小文字を保持し、タグラベルクリックで編集できるように変更。タグフィルタバーを折り畳み可能なバーとして実装。

**当初の抜け**
- タグ入力時に小文字強制変換していた。
- 削除のみで編集ができなかった（ラベルクリックで入力欄に戻す仕組みを後から追加）。

---

### 5. タブの全閉じ・フォルダ切替時のタブ復元

「全て閉じる」ボタンと、フォルダを切り替えて戻った時に開いていたタブを復元する機能。

**実装ポイント**
- `localStorage` にフォルダパスをキーとしてタブ状態を保存。

---

### 6. 検索フィルタの改善（空フォルダ非表示・深いディレクトリ対応）

検索時にマッチしたファイルを含まないフォルダを非表示にし、遅延レンダリングされた深い階層のファイルも検索対象に含める。

**当初の抜け**
- 遅延レンダリング（`_dirChildren` プロパティで未描画の子を保持）の存在を検索時に考慮していなかった。深い階層のファイルが検索結果に現れなかった。
- `isFiltering` フラグで検索・タグフィルタ・フラグフィルタを統合し、未描画ディレクトリを do-while で強制レンダリングして解決。

**AI として不足していた視点**
- 遅延レンダリングを実装した時点で、検索との整合性を同時に考慮できていなかった。最適化と機能の相互作用を見越す視点が弱かった。

---

### 7. ファイル管理（作成・リネーム・削除）

右クリックコンテキストメニューからファイル・フォルダの作成・リネーム・削除を実装。

**当初の抜け・バグ**

#### コンテキストメニュー後の blur 競合
- コンテキストメニューの mousedown 後に `input.focus()` を即時呼び出すと、続く click イベントのフォーカス移動と競合してインライン入力が即座に消えた。
- `setTimeout(() => input.focus(), 0)` で次のイベントループに遅延させて解決。

#### .md 拡張子がクリップされる
- `.file-name` の `overflow: hidden; white-space: nowrap;` により、拡張子ラベル `.inline-input-ext` がはみ出した分が非表示になった。
- リネーム中に `nameEl.style.overflow = 'visible'` を設定し、終了時に戻すことで解決。

#### タグがリネーム後にフィルタされない
- ファイルリネーム時は `state.tags` と `tags.json` のキー更新を実装していたが、フォルダリネームでは配下のファイルパスが一括で変わることを見落としていた。
- `renameFileTags()` をプレフィックス一致の一括置換に修正。

#### Windows のパス区切り文字不一致
- `tags.json` のキーはクライアントの `toRelative()` がバックスラッシュで生成するが、`renameFileTags()` がスラッシュ正規化後のキーで書き込んでいたためミスマッチが発生。
- 新しいキーはオリジナルのパス区切り文字をそのまま引き継ぐように修正。

**AI として不足していた視点**
- ファイル単体のリネームを実装した時点でフォルダリネームへの波及を考慮できなかった。
- Windows のバックスラッシュと forward slash の混在は OS 依存の問題であり、クロスプラットフォームへの意識が初期実装で欠けていた。

---

### 8. ルートフォルダへの作成導線

ツリーにルートフォルダ行を追加し、直下へのファイル・フォルダ作成を可能にした。

**当初の抜け**
- 既存のコンテキストメニューはサブフォルダ行にしか付けておらず、ルート直下への作成手段がなかった。

---

### 9. フォルダ作成時の空フォルダ問題

ファイルスキャンが Markdown ファイルを含まないフォルダをツリーに表示しない仕様のため、空フォルダを作成してもツリーに現れなかった。

**対処**
- フォルダ作成時に同名の空 `.md` ファイルを自動生成し、フォルダが即座にツリーに現れるようにした。
- インライン入力にヒント「(xxx.md が自動作成されます)」をリアルタイム表示。

**AI として不足していた視点**
- フォルダ作成機能を実装した時点で、スキャン仕様との整合性を確認できていなかった。既存の設計制約を横断的に把握して新機能に反映する意識が弱かった。

## コードレビューで指摘された問題（sandbox/review.md）

外部レビューにより、AI 実装では見落としがちな以下の問題が発見された。

### activeTabPath の境界チェック漏れ

```js
// 修正前（兄弟パスに誤マッチ）
if (state.activeTabPath?.startsWith(oldPath)) ...

// 修正後（セパレータ境界を確認）
if (state.activeTabPath === oldPath ||
    state.activeTabPath?.startsWith(oldPath + '\\') ||
    state.activeTabPath?.startsWith(oldPath + '/')) ...
```

`/a` をリネームすると `/ab.md` まで誤更新される可能性があった。`state.tabs` の更新条件はセパレータ境界を見ていたが、`activeTabPath` だけ条件が甘かった。

### リネーム先・フォルダ作成先の存在確認なし

サーバー側でリネーム先や作成先の存在確認をしていなかった。OS 依存の生エラー（`EEXIST` など）がそのままクライアントに返っていた。`access()` による事前チェックを追加し日本語エラーに変換。

### エラー時に画面が古いまま残る

FS 操作とタグ更新・UI 更新が直列しており、タグ更新で例外が発生すると `refreshTree()` や `openFile()` に到達しないまま catch で終了していた。catch ブロックでも `refreshTree()` を呼ぶようにして画面を最新化。

## AI 協働開発における共通の傾向

以上の実装を通じて見えてきた AI の弱点をまとめる。

### 1. 最適化と他機能の相互作用を見逃しやすい
遅延レンダリング・スキャン最適化などを実装した時、その変更が別機能（検索・タグフィルタ）に与える影響を同時に考慮できていなかった。

### 2. OS 依存の挙動への意識が薄い
パス区切り文字（`\` vs `/`）、ファイル操作の OS ごとの差異など、クロスプラットフォームの問題は実際に動かすまで顕在化しにくい。

### 3. 既存の設計制約を横断的に把握できない
新機能を追加する際、既存の仕様（スキャンが空フォルダを除外する、遅延レンダリングで未描画の子がある、など）との整合性確認が不十分になりやすい。

### 4. 「境界条件」の確認が甘い
文字列の prefix 一致チェックにおけるセパレータ境界、存在しないパスへの操作、部分失敗時の状態整合など、エッジケースの網羅が弱い。

### 5. ユーザーが実際に使うまで分からない問題がある
パフォーマンス（5 秒のスキャン）、フォーカスの競合（blur 即時発火）、視覚的な見づらさ（フォントサイズの同色化）など、動かして初めて分かる問題は AI 単独では発見しにくい。
